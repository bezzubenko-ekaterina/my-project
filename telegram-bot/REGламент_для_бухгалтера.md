# Регламент для бухгалтера (Telegram-бот)

## 1) Что нельзя менять

1. Не переименовывать листы: `Clients`, `Payments`, `NotificationLog`.
2. Не переименовывать заголовки колонок.
3. Не удалять первую строку (заголовки).
4. Не удалять исторические строки платежей: менять статус (`paid` / `cancelled`), а не удалять.

---

## 2) Новый клиент (одна организация)

1. На листе `Clients` добавить новую строку:
   - `client_id` (уникальный, например `C010`)
   - `name`
   - `org_type`
   - `inn`
   - `contact_person`
   - `link_code` (уникальный, 6 символов)
   - `active` = `1` (или `TRUE`)
2. Колонки `telegram_id` и `linked_at` руками не заполнять.
3. Отправить клиенту его `link_code`.
4. Клиент в боте отправляет код (`/start КОД` или просто `КОД`).
5. Проверить, что бот заполнил `telegram_id` и `linked_at`.

---

## 3) Если у клиента несколько организаций

1. Для каждой организации в `Clients` должна быть отдельная строка.
2. У каждой строки:
   - свой `client_id`,
   - свой `link_code`.
3. Клиент отправляет в бота каждый `link_code` по очереди.
4. После каждой привязки бот показывает список всех организаций, привязанных к этому Telegram-аккаунту.

Важно: один и тот же `link_code` нельзя использовать для разных организаций.

---

## 4) Как вносить платежи

На листе `Payments` для каждого платежа заполнять:

- `payment_id` (уникальный)
- `client_id` (должен совпадать с `Clients.client_id`)
- `tax_type`
- `period`
- `amount` (если пусто, уведомления не отправляются)
- `due_date`
- `status`
- `amount_confirmed_at` (когда сумма подтверждена)
- `paid_at` (когда реально оплачено)
- `notes` (по необходимости)

---

## 5) Значения статуса и их смысл

- `pending` — платеж активен, напоминания разрешены.
- `draft` — сумма/данные в работе, клиенту отправляется "Ваш бухгалтер активно считает..." по кнопке.
- `paid` — оплачен, напоминания не отправляются.
- `cancelled` — отменен, напоминания не отправляются.
- пустой статус — трактуется как "в работе" (как `draft` для кнопки).

---

## 6) Когда бот отправляет уведомления клиенту

Условия отправки любого уведомления:

1. `status = pending`
2. `amount` заполнен
3. у клиента есть `telegram_id` (клиент привязан)

Типы уведомлений:

1. За 7 дней до `due_date`
2. За 3 дня до `due_date`
3. За 1 день до `due_date`
4. `amount_confirmed` — когда `amount_confirmed_at` непустой

Дополнительно:

- Бот проверяет таблицу циклически (по `CHECK_INTERVAL_SECONDS` на сервере).
- Уведомления отправляются начиная с часа `NOTIFICATION_HOUR` (локальное время сервера по `TIMEZONE`).
- По одному `payment_id` и типу уведомления повторная отправка не делается (антидубль через `NotificationLog`).
- Если в момент проверки нет `telegram_id`, в `NotificationLog` пишется `skipped_unlinked`.

---

## 7) Что проверять после любых изменений

1. В `Clients` после привязки есть `telegram_id` и `linked_at`.
2. В `Payments` у активных платежей (`pending`) заполнен `amount`.
3. В `NotificationLog` появляются записи со статусами `sent` / `skipped_unlinked` / `failed`.

---

## 8) Частые ошибки

1. В `Payments.client_id` не совпадает с `Clients.client_id` -> уведомления не уйдут.
2. `amount` пустой при `pending` -> уведомления не уйдут.
3. Клиент не отправил `link_code` -> `telegram_id` пустой -> `skipped_unlinked`.
4. Переименовали колонки/листы -> бот перестает корректно читать таблицу.
